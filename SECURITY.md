Security Model and Protocol Design

Overview
This document explains the design philosophy, protocol mechanics, and security assumptions behind SecureEthUsdcSwapV3.
The contract is built to operate safely in adversarial blockchain environments where attackers may exploit mempool visibility, transaction ordering, or economic incentives.

Design Goals
The protocol aims to achieve three properties:

1. Trustless exchange
2. Economic safety
3. MEV resistance

Users should be able to swap assets without trusting each other and without relying on centralized intermediaries.

Threat Model
The protocol assumes the following threats exist:
• MEV bots monitoring the mempool
• malicious swap participants
• transaction ordering manipulation
• griefing attempts
• denial‑of‑service attacks
• timestamp manipulation by validators
The design focuses on neutralizing these threats using cryptographic and economic mechanisms.

Core Protocol Mechanics
The protocol uses a hash‑locked swap mechanism.
A secret is generated by the initiator:
secretHash = keccak256(secret, nonce)
The secret is revealed only during swap completion, enabling atomic asset exchange.

Commit‑Reveal System
Without protection, a swap initiation transaction could be copied from the mempool.
An attacker could submit the same parameters with higher gas and hijack the swap.
To prevent this, the protocol uses a commit‑reveal scheme:
Step 1 — Commit
A hash of all swap parameters is submitted.
Step 2 — Reveal
Parameters are revealed and verified against the commitment.
The commitment includes:
• secretHash
• nonce
• counterparty
• ethAmount
• usdcAmount
• duration
• initiator address
• salt
Because the initiator address is part of the commitment, only the original sender can reveal it.

Symmetric Collateral Model
The protocol implements a mutually assured destruction (MAD) incentive model.
Both participants lock collateral:
• Initiator: 5% of ETH amount
• Counterparty: 5% of USDC amount
If the initiator fails to reveal the secret:
• both collaterals are burned
• the counterparty receives the premium
• principals are returned
This structure discourages griefing and collusion.

Premium Mechanism
The initiator pays a premium.
This premium compensates the counterparty if the initiator abandons the swap and acts as time‑lock insurance for capital commitment.

Rate Limiting
To prevent spam attacks and griefing loops, the protocol enforces:
• maximum active swaps per address
• cooldown between swap creation
• cooldown between funding operations
These limits make automated griefing economically inefficient.

Reputation Model
The protocol tracks behavioral history for each address:
• completed swaps
• refunded swaps
• griefed swaps
• cancelled swaps
A derived score provides a public signal of past behavior.
Reputation does not act as a hard enforcement mechanism.
Users with low scores are not automatically blocked from initiating swaps.
Reputation functions as:
• observability
• transparency
• counterparty signaling

Collusion Detection
Repeated griefing patterns between two addresses may indicate collusion.
The protocol applies a heuristic:
If either:
• the initiator griefed ≥ 2 times
• the counterparty has ≥ 2 refunded swaps
the pair may be permanently flagged.
Flagged addresses cannot swap with each other again.
This mechanism is heuristic‑based and not cryptographic proof.
False positives are theoretically possible.
The threshold is intentionally conservative.

Timestamp Drift Protection
Validators can manipulate block.timestamp slightly.
To mitigate this risk, the protocol applies a drift buffer to all deadline checks.
This prevents a validator from invalidating a legitimate secret reveal by pushing timestamps marginally forward.

Pull‑Payment Pattern
Direct ETH transfers can fail if the receiver rejects ETH.
To avoid denial‑of‑service scenarios, the contract uses pull payments.
ETH owed to users is stored in:
pendingWithdrawals[address]
Users must explicitly withdraw their funds.

Finite State Machine
The protocol enforces strict state transitions:
EMPTY → INITIATED → FUNDED → COMPLETED
Alternative transitions:
INITIATED → CANCELLED
INITIATED → REFUNDED
FUNDED → REFUNDED
Invalid transitions are rejected by the contract.

Atomicity Clarification
The protocol implements an economically atomic swap.
While the lifecycle spans multiple transactions, the design ensures:
• unilateral asset capture is impossible
• dishonest behavior is penalized
• principal recovery is enforced
Atomicity is enforced at the protocol and economic level rather than the single‑transaction EVM level.

Security Assumptions
The protocol assumes:
• Ethereum cryptography remains secure
• keccak256 collision resistance holds
• validators cannot manipulate timestamps beyond allowed drift
• users protect their private keys
Under these assumptions, the protocol ensures safe peer‑to‑peer swaps.

Conclusion
SecureEthUsdcSwapV3 combines cryptographic commitments, economic incentives, and strict state enforcement to create a robust trustless swap protocol.
The design aims to make dishonest behavior economically irrational while enabling honest users to exchange assets safely without intermediaries.








## Reporting a Vulnerability

If you discover a security vulnerability in this repository or in the
SecureEthUsdcSwapV3 smart contract, please report it responsibly.

### How to report
  We aim to acknowledge reports within 72 hours.
- Send a detailed report to: **asperant001@gmail.com**
- Include:
  - contract address
  - network (Base Mainnet or Sepolia)
  - description of the issue
  - potential impact
  - steps to reproduce (if applicable)

Please do **not** open public GitHub issues or disclose the vulnerability
publicly before it has been reviewed.

## Bug Bounty

At this time, there is **no active bug bounty program**.

Valid vulnerability reports will be acknowledged and reviewed in good faith.

Please do not perform denial-of-service attacks or exploit vulnerabilities
against production systems.
